A browser-based maze/monster-capture game built with React, TypeScript, and webpack. Full design rationale and decisions live in `PLAN.md`.

## What it does

The player navigates a 2D grid maze from a top-down perspective by clicking cells, in straight lines only (no diagonals, no jumping over walls). The maze holds 39 monster nodes plus one goal tile (39 + 1 = 40, a nod to the "40 years of life" framing); an uncaptured monster (or the goal) blocks its cell like a wall. Tapping one from a distance walks the player up to it; walking directly adjacent to a monster starts a short conversation, which transitions into a real-time battle — the player attacks with their already-captured monsters (each on its own cooldown) until the wild monster's HP hits 0 (captured) or the player's HP hits 0 (returned to the map, monster stays uncaptured, retryable anytime). The goal tile also leads into a conversation, but its battle stays locked until every monster is captured; once unlocked, that conversation ends into a battle against the goal itself, followed by a finale conversation on a win. Every captured monster trails behind the player on the map as a small "duckling train."

## Current state

Core loop is implemented end-to-end: map traversal, capture-gated blocking, conversation, real-time battle, and persistence. Content is a first draft, not final: the 39 monster definitions and their conversation scripts were bulk-generated from source material and are meant to be hand-revisited (see `PLAN.md`'s Open Questions).

Every image in the repo (all 39 monster icons, the goal sprite, the player's 3 map sprites) is a real `.png` file, imported normally in code - webpack's `url-loader` (see `webpack.config.js`, `test: /\.(png|svg)$/` - `.svg` supported too even though nothing currently uses it) always inlines them as base64 data URIs at build time regardless of size, so the final deployed app still ships as one self-contained JS bundle - the GAS web app only serves a single HTML file, there's no separate static asset serving.

**This is deliberate, not an oversight - don't "fix" it by switching to file-referenced assets.** Apps Script has no project file type for binary assets at all (`.clasp.json`'s `htmlExtensions`/`scriptExtensions`/`jsonExtensions` are the only file kinds clasp will push - there's no binary equivalent), and `doGet` can only return `HtmlOutput`/`TextOutput`, so even if the bytes lived in the project there'd be no way to serve them back with an `image/png` content-type from this same web app. The only ways to *not* inline images would be hosting them on an external service (Google Drive, a CDN, GitHub raw, ...) - a real external dependency (uptime, permissions, link format) - or a lazy-loaded `doGet(?asset=id)` endpoint that still ships the same base64 bytes, just deferred past first load. Neither has been worth the added complexity: the built `gas/index.html` measured 740KB as of 2026-07-17 (39 icons + sprites = 456KB on disk, base64's ~33% overhead accounts for the rest), well under any GAS payload limit and an unremarkable size for a web page. Revisit only if a real symptom shows up (slow load, hitting an actual GAS limit) - not just because the number looks big for "one file."

### Global scale

`src/scale.ts` exports a `SCALE` constant (currently `3`) and sets it as the `--scale` CSS custom property on `<html>`. All sizing derives from this:
- JS: `CELL_SIZE = 100 * SCALE`
- CSS: `font-size: calc(13pt * var(--scale))`, dialog height `calc(120px * var(--scale))`, etc.

To make the UI bigger or smaller, change only `SCALE` in `scale.ts`.

### Game state & persistence

The persisted slice (`store/gameStore.ts`) holds only: player map coordinate, the single cell the player last moved from (`previousPosition`, `null` until the first move ever - seeds the duckling trail below on load, `setPosition` overwrites it with the pre-move `position` on every move; `teleportTo` instead sets it to the *same* cell as the new `position`, since a teleport was never walked from anywhere - see `MiniMap`'s own tap-to-teleport below), the sprite's last facing direction (`"left" | "right" | "up" | "down"`, updated automatically by both `setPosition` and `teleportTo` - shared as `facingFromMove` - based on whichever axis actually changed between the old and new position, x taking priority over y; ordinary movement only ever changes one axis, but a mini-map teleport can be diagonal, so this is really "whichever axis differs, x first" rather than a guarantee only one axis changed), captured monsters (id -> ISO capture date), `monsterOrder` (the front-to-back id order shared by the duckling trail and the battle attack line - see "Family-adjacency attack bonuses"), per-attack cooldowns (monster id, or `"innate"` for the protagonist's own attack -> next-available timestamp in ms), per-encounter battle-loss cooldowns (`battleCooldowns`, monster id -> next-challengeable timestamp in ms - never `"goal"`, which is exempt from this lock entirely - see "Losing locks that battle out for a while" under "Conversation -> Battle flow"), explored cells (see "Fog of war" below), and `goalDefeatedAt` (ISO date of the player's first win of the goal battle, or `null` - see "Conversation -> Battle flow"). Everything else — conversation progress, in-battle HP, which page/screen is showing — lives in the ephemeral `store/flowStore.ts` and is never persisted, so a reload mid-conversation or mid-battle just drops back to the map with no side effects (cooldowns already spent still apply).

On first launch (or whenever no key is present), `StateKeyGate` blocks rendering: in local dev it silently assigns `"local-dev"`; in the deployed GAS environment it prompts the player for a save-state key. The key itself lives in the URL's `?key=` query string, not `localStorage`, so it survives a cleared browser, a different device, or a bookmarked/shared link. The deployed app runs inside an iframe on its own `googleusercontent.com` origin (standard GAS behavior for every web app, unrelated to this codebase), so client-side `window.location` there is that iframe's own src, never the visible `/exec?key=...` URL the player actually sees — `persistence.ts`'s `getStateKeyFromUrl()`/`setStateKeyInUrl()` route around this rather than reading/writing `window.location` directly. Reading: `gas/Code.js`'s `doGet(e)` still receives the outer request's real query string server-side, so it renders `index.html` as an `HtmlService` template (`createTemplateFromFile`, not `createHtmlOutputFromFile`) and bakes `e.parameter.key` in as a `window.__STATE_KEY__` global (plus `window.__WEBAPP_URL__` from `ScriptApp.getService().getUrl()`, this deployment's own canonical `/exec` URL) — `getStateKeyFromUrl()` reads that global first, falling back to a plain `window.location.search` read only for local dev, where nothing injects it. Writing: the iframe has no access to its cross-origin parent's history/address bar at all, so `setStateKeyInUrl()` forces a real top-level navigation (`window.top.location.href` built from `window.__WEBAPP_URL__` + the new key) when framed — a full page reload, but only on this rare explicit action (first-run prompt, `/settings`), never on ordinary gameplay saves; unframed (local dev, or a direct visit to the raw content URL) it still does the old in-place `window.history.replaceState`. `webpack.config.js`'s `templateContent` only emits the `<script>` scriptlet (using GAS's `<?!= ?>` unescaped-printing syntax) for production builds — local dev has no template evaluation step to resolve it. The two navigation spots that switch between `/` and `/settings` (`Settings`'s close button, `Game`'s ⚙ link) explicitly carry `window.location.search` forward so react-router's own path-only `history.push`/`<Link to="/settings">` doesn't drop whatever query string is already there. The key is then used to look up/save a single JSON blob (position + captured + cooldowns + a timestamp) to `localStorage` (for instant reload, still scoped `gameState:<key>`, not a single shared slot — critical for `/settings`: changing the key calls `hydrate()` for the *new* key, and since its local cache starts genuinely empty (or reflects only what this device has saved under that specific key before), a fresh remote snapshot is never crowded out by another key's stale local data) and to the Google Sheet (via `google.script.run`, guarded by `typeof google !== "undefined"` so local dev is unaffected). Within the same key, `resolveHydratedState` (`store/persistence.ts`) picks whichever of local/remote has the newer timestamp.

Remote saves go through `store/remoteSync.ts`'s `createRemoteSync` — a small controller (DI'd with the actual `saveRemoteState` call) that tracks at most one not-yet-synced write at a time. A failed save (offline, GAS error, etc.) is never retried in a busy-loop; it just stays pending until something external asks again — `gameStore` wires that up via the browser's `online` event plus a 30s fallback timer, so a write made while offline reaches the Sheet once connectivity returns, without the player having to do anything. A newer write always supersedes an older still-pending/in-flight one. Switching state keys (`setStateKey`) calls `clearPending()` first, so an unsynced write for the *old* key doesn't linger and get retried against the wrong slot — it's still safely cached in that old key's local storage, just not pushed further until the player switches back and changes something again.

### Fog of war (mini-map)

`gameStore.exploredCells` is a `Record<"x,y", true>` (see `components/Maze/exploration.ts` for the `cellKey`/`computeTraversedCells`/`revealCells` helpers), persisted like everything else. Every `goto` move — including the multi-cell slides this game already supports, not just single steps — reveals every cell along the traversed line, not merely the destination. `MiniMap` renders any cell missing from this set as solid fog regardless of what's actually there (wall/road), so the overview fills in only as the player actually walks past that area; the player's own current cell is always shown regardless. Every uncaptured monster and the goal tile are exceptions - they punch through the fog as beacons so the player always knows *something* is there, even in unexplored territory, since finding the actual path there on foot is the point. Only `MiniMap` respects fog — the main `Maze` view always shows everything.

`MiniMap` is also tappable: an already-explored cell that's currently passable (`Maze/passability.ts`'s `isPassableCell`, shared with `Maze`'s own `goto`/`findStoppingPoint` - plain road, or a captured monster's now-road-like cell, never a wall, an uncaptured monster, the goal tile, or unrevealed fog) teleports the player straight there on tap (`gameStore.teleportTo`, which sets both `position` and `previousPosition` to that same cell - see above), skipping the maze's usual straight-line-only walk. Gated on `flowStore.mode === "map"`, same as `goto`. Since a teleport can land anywhere (not just along the player's shared row/column), it can't extend the duckling follower trail the normal step-by-step way (`followerTrail.ts`'s `extendTrail` assumes a contiguous walked path) - instead `Maze` derives `isTeleportedInPlace` (`previousPosition` equal to the current cell - the one signature `teleportTo` leaves that an ordinary walked step never does) and collapses the trail onto the new cell outright the moment that flips true, exactly like entering the goal's house (see below); ordinary walking right after re-extends it from that single point same as leaving the house does. Being derived straight from persisted state rather than an ephemeral counter, this also self-corrects for free on a reload that lands mid-teleport (no live "just teleported" event to miss).

That single-cell collapse also empties `followerPoints` (`resamplePath` needs at least 2 cells), so every follower would otherwise fall back to whichever point `lastPassableFollowerPointRef` last held (normally updated only when the cell behind the player's current facing is actually passable - see the "duckling train" comment below, which a teleport destination has no guarantee of). `noPathFollowerPoint` instead forces the player's own cell center whenever `isTeleportedInPlace` (or `houseState === "occupied"`) is true, snapping the whole train to stand right on the player unconditionally - a teleport has no "direction arrived from" for the usual half-cell-behind offset to trail from, so unlike the walking fallback this doesn't special-case whether the behind-cell happens to be passable.

### Monster blocking, capture

`Maze` scans `map.txt` top-to-bottom/left-to-right; the *n*th `M` it finds is `MONSTERS[n]`. An uncaptured monster cell blocks movement like a wall — it's still a valid tap target from anywhere along a clear straight line, but `goto` only starts the encounter (`flowStore.startEncounter`) when the player is already adjacent; tapping it from further away instead walks the player up to the cell just before it, same as approaching any other obstacle. Tapping any cell whose straight-line path is blocked partway (by a wall, an uncaptured monster, or the goal) - not just the obstacle itself - now walks the player as far as `findStoppingPoint` (`exploration.ts`) says is actually reachable, stopping just short of the first blocker, rather than refusing the move outright. A captured monster's cell is just a normal road from then on. Every monster can be challenged at any time — there's no unlock condition or time window of any kind.

### The goal tile

The map's single `'F'` cell (found via `Maze/goalPosition.ts`'s `findGoalCell`) works like a monster for movement purposes — blocks like a wall, walk-up-then-tap same as above — and, like a monster, its conversation can lead into a battle (`flowStore.startGoalEncounter`, a separate ephemeral flag from `activeMonsterId` since there's no specific monster involved — `Battle`/`ConversationView` key off `isGoalEncounter` wherever they'd otherwise key off `activeMonsterId`). `ConversationView` picks the goal's pages by state: `GOAL_HINT_CONVERSATION` while any monster is still uncaptured (ends in `"end"`, no battle - the battle stays locked until `captureLogic.isFullyCaptured`); `GOAL_CHALLENGE_CONVERSATION` once fully captured but the goal battle hasn't been won yet (ends in `"enter_challenge"`, same as a monster's terminal page); and, once a `battleOutcome` comes back from that battle, `GOAL_FINAL_CONVERSATION` (win), `buildGoalLossConversation(GOAL_NAME)` (lose - see below), or the generic `buildOutcomeConversation(GOAL_NAME, outcome)` reaction (escape only) - mirroring how a monster encounter picks its post-battle text. Any `"monster"`-speaker page renders the goal's own sprite/name (`data/goalEncounter.ts`'s `GOAL_NAME`, plus `assets/goalSprite.png`) in place of a monster's. The goal isn't "captured" away like a monster - the tile stays enterable indefinitely, so the challenge conversation and battle can be replayed (see the `跳過戰鬥` skip button below).

Unlike a monster, the goal's own art is fixed facing left rather than mirroring to face the player from any side - so `Maze/index.tsx`'s `goto` only calls `startGoalEncounter` when the player is directly to its left (`r === y && c > x`, the only case where the goal is actually "looking at" the player); approaching from the right, above, or below still blocks movement like a wall, it just never starts a conversation. Once `gameStore.goalDefeatedAt` is set, a few 💕 rise from the goal's sprite like drifting smoke on a permanent, looping fade in/out (`Maze/styles.css`'s `.loveSmoke`) - a standing "challenged and defeated" marker, not tied to any particular visit. Alongside it, a pulsing neon-glow "40" (`.goalNeonSign` - a nod to the game's own 40th-birthday framing, see the top of this file) sits mounted above the house's roof - same `goalDefeatedAt !== null` trigger and same "renders regardless of houseState" placement (a sibling after the occupied/empty branch, not inside either arm of it) as the hearts, so it shows in both the "empty" and "occupied" house states identically. A third effect on the same trigger, a few fireworks launching from the house's own center up to around the "40" sign, is dot-based rather than an emoji glyph - each instance is a rising `.fireworkRocket` dot plus a `.fireworkBurst` of `.fireworkSpark` dots that fly outward along their own fixed `--spark-angle` (`rotate()` then `translateY()` composing to send each spark along that rotated axis), all three timed off one shared `--firework-delay` custom property so the rocket and its own burst never drift out of sync with each other.

#### The house

Once the goal's been defeated at least once, its cell doubles as a house (`Maze/houseState.ts`'s `computeHouseState`, a pure `(goalDefeatedAt, position, goalCell) -> "none" | "empty" | "occupied"`, unit-tested in `houseState.test.ts`). There's no separate "inside the house" flag persisted anywhere - since the player's `position` is already what's persisted, "occupied" is derived purely from the player's own cell coinciding with the goal's, and reloading mid-visit lands back in the same state for free. `"none"` (goal never defeated) renders nothing extra. `"empty"` (defeated, player elsewhere) underlays `assets/home-empty.png` behind the normal goal sprite - the goal tile still blocks movement and still starts `GOAL_CHALLENGE_CONVERSATION` on approach, same as before this feature existed. `"occupied"` (defeated, player standing on the goal's own cell) swaps both the goal sprite *and* `MazeContainer`'s player pin out for a single combined `assets/home.png` (house + player + goal all in one image) - `MazeContainer` computes the same `computeHouseState` result itself purely to decide whether to hide its pin, since Maze's own cell rendering already draws the combined art in that case.

The only way `position` ever actually becomes the goal cell (normal movement can't reach it - the tile blocks like a wall) is `ConversationView`'s `advance()` calling `gameStore.setPosition` to the goal's coordinates right as it also calls `endEncounter()` on the finale conversation's own terminal ("end") page - i.e. only once the player has read that whole win conversation, not the instant the battle itself is won (`recordGoalWin`, in `Battle/index.tsx`, fires earlier and independently - it's what makes `GOAL_FINAL_CONVERSATION` the pages shown at all, but doesn't move the player). That's the "the player sprite enters the house" moment. From there the player can walk back out left or up (both roads on the map bordering the goal tile; right and below are walls) via completely ordinary movement, since `goto`/`findStoppingPoint` only ever check the *destination* cell's passability, never the cell being walked away from. Leaving drops `computeHouseState` back to `"empty"`; re-entering requires talking to the goal tile again (i.e. winning - or skipping via `跳過戰鬥` - another goal battle).

The house sprites sit at `z-index: 500` (`Maze/styles.css`) - above the duckling follower trail (whose own z-index tops out at the captured count) so the train always passes behind the house, and below the player pin's `1000` (moot when occupied, since the pin is hidden outright then) *unless* the player is standing in the row above the goal cell. They're 120% of the cell's own size, horizontally centered but anchored to `bottom: 28%` rather than the cell's literal floor or its vertical center - both source images' own house art already touches their canvas's bottom edge with no margin, so either of those naturally-simpler anchors left the house sitting lower than every other sprite here; `28%` instead lines it up just above the *top* of `.footShadow`'s own ellipse (`bottom: 6% + height: 16%`), roughly where a sprite's feet actually meet its shadow. That same overhang is also why the roof needs to draw *in front of* the player rather than behind it whenever the player is north of the house: `houseState.ts`'s `isAboveGoalCell(position, goalCell)` (true whenever the player's row is smaller than the goal's - row index increases downward) drives a `.pinBehindHouse` class on `MazeContainer`'s own pin, dropping its z-index to `100` (below the house's `500`/`501`/`502` tiers) only in that case - every other row keeps the pin at its usual `1000`, on top of the house.

Since entering the house is a teleport (`ConversationView`'s `setPosition` above) rather than a walked step, it never goes through `goto`'s own `extendTrail` call - so `Maze/index.tsx` watches `houseState` itself and, the moment it flips to `"occupied"`, *resets* the follower trail to just the house's own cell (rather than extending it). `resamplePath` needs at least 2 cells to place anyone individually, so with only one cell in the trail every follower falls back to the same shared point - the whole duckling train collapses onto that single point, i.e. the entire train (head to tail) visibly enters the house alongside the player, not just the closest few. That shared fallback is `noPathFollowerPoint`, which while occupied is the goal cell's own dead center (`playerCenter`) rather than the usual `fallbackFollowerPoint` (half a cell behind the player) - the latter would leave the train hugging the doorway edge instead of sitting inside with the player. Leaving is then an entirely ordinary `goto`/`extendTrail` walk starting fresh from that single-cell trail: only as many followers as the walked-so-far path can actually fit get their own resampled point, and the rest stay clamped to the path's last point (still back at the house) - so the tail visibly lags behind, still "in the house," until the player's put enough distance between them for the whole train to have emerged and resumed its usual spread-out trailing.

The trail's own initial `useState` seed mirrors this same single-cell collapse for a save loaded *already* mid-occupancy (a save made while standing in the house, then reloaded fresh) - without that, the seed would fall back to its normal `[[x,y], previousPosition]` shape, and since entering the house is a teleport, `previousPosition` there is still whatever cell the player last walked in from (outside the house), spreading the train between the goal cell and that outside cell on load instead of bunching it at the goal cell like a live entry does.

Tapping the player's own cell is normally a no-op in `goto` (there's nowhere to walk), but while `houseState === "occupied"` that tap instead calls `flowStore.replayGoalFinale()` - a `startGoalEncounter`-like action that sets `isGoalEncounter: true` *and* `battleOutcome: "win"` directly, so `ConversationView` picks `GOAL_FINAL_CONVERSATION` immediately rather than `GOAL_HINT_CONVERSATION`/`GOAL_CHALLENGE_CONVERSATION` (which is what plain `startGoalEncounter` would show, since those only fire when `battleOutcome` is `null`). It also forces `isFirstGoalWin` to `false` - reaching the house at all requires having already gone through the finale once, so a replay is never genuinely the first viewing, but `isFirstGoalWin` itself stays `true` for the rest of the session after the real first win (nothing else ever resets it), so skipping this would keep wrongly hiding `跳過對話` on every replay after the first. This replays the whole finale on demand, without another real battle - its own terminal page still ends by calling the same `enterHouseIfWinner` (`setPosition` to the goal cell, itself already a no-op here, then `endEncounter`), so leaving afterward works exactly like a fresh win.

### Conversation -> Battle flow

Walking into a monster shows its portrait conversation (`Dialog`/`ConversationView`) — linear pages alternating between the protagonist (小風) and the monster, tap to advance. While a page is being typed out, `ConversationView` writes the current speaker into `flowStore.talkingSpeaker` ("protagonist" | "monster" | null); `Maze` reads that to play a small looping "jump" animation on the actual map sprite that's talking (the player's sprite, or the active monster's icon) rather than animating the dialog's own portrait image. The terminal page's `enter_challenge` action computes the wild monster's max HP from the player's current capture count (`computeWildMaxHp`) and switches `flowStore.mode` to `"battle"`, which swaps `Maze`/`Dialog` out for the full-screen `Battle` component. Battles are real-time: the protagonist (left, HP above its sprite) and the wild monster (right, HP below its sprite) each show their own name/HP paired only with their own sprite; the player can tap any captured monster (plus their own innate attack) to attack - see "Family-adjacency attack bonuses" below for how much damage that actually deals and what else it triggers - each on an independent 1-minute cooldown persisted in `gameStore`; the wild monster automatically deals 1 damage every 10 seconds; healer monsters (~5% of the roster) restore HP instead of dealing damage; an escape button leaves the battle at any time - a real escape (`flowStore.concludeBattle("escape")`) above half HP, but below half HP it's treated as a genuine loss instead (`setPendingOutcome("lose")`, same sink animation, battle-loss cooldown, and outcome conversation a real defeat gets), so a player can't just bail out right before actually losing to dodge the cooldown lock. Attacking/being-hit trigger a brief lunge/knocked-down-stumble bump fired directly on the sprite's DOM node via the Web Animations API (`Battle/index.tsx`'s `playBump`/`attackBumpKeyframes`/`hitBumpKeyframes`), rather than a React-state-driven CSS class - the enemy also gets its own ground shadow (`.enemyShadow`, same proportions as the map's `.footShadow`) that's bumped in lockstep with it (same `attackBumpKeyframes` call, same duration) whenever the enemy itself lunges to spit, so it stays under the sprite's feet through that horizontal shift rather than sitting fixed while the sprite visibly moves away from it; it's *not* bumped during the enemy's own vertical hit-knockback, since `hitBumpKeyframes` has no horizontal component for the shadow to mirror there; healing instead triggers a `filter`-based glow that's still class/state-driven (`useHealEffect`, `.playerHeal`), since it's the only effect animating that property. Splitting it this way means a hit landing mid-heal no longer cuts the glow short — a CSS `animation` shorthand can only resolve to one value per element, so two className-driven animations fight over the same slot, but a WAAPI animation and a CSS-class animation are independent mechanisms (and here also animate different properties: transform vs filter), so they run concurrently with no conflict. However the battle ends — win (`flowStore.concludeBattle("win")`), lose, or escape — `mode` returns to `"conversation"` with `battleOutcome` set, so `ConversationView` shows a short generated reaction (`buildOutcomeConversation`) before actually dropping back to the map (`endEncounter`, which also clears `battleOutcome`). A win only actually calls `captureMonster` once the post-battle pause+fade finishes (not the instant HP hits 0), so the just-defeated monster doesn't show up in the attack grid while its own defeat is still playing out.

`Battle` always shows "贏"/"輸" dev buttons once `isDevStateKey(stateKey)` is true (`store/devMode.ts` - any save key containing `"peteranny"`, not just an exact match) - no separate toggle gates them, unlike `devReleaseEnabled`/`devCooldownLockDisabled` below. The same gate also attaches the exact `hp / maxHp` numbers (`.hpDevValue`) beneath each side's `HpBar`, for reading precise values instead of eyeballing the bar's fill percentage. Each just deals lethal damage through the normal `damageWild`/`damageProtagonist` path, so they behave exactly like a real killing blow (capture, fade, outcome conversation, all included) rather than a separate force-win code path. The map screen's own compact icon row (beside the ⚙ link, `Game/index.tsx`) shows the same dev gate for three more affordances: `devReleaseEnabled` (✕, a real toggle - click a duckling-trail follower to un-capture it, see `Maze/index.tsx`), then either a "🔓" toggle (`devCooldownLockDisabled` - bypasses `battleCooldowns` entirely while on, exactly like `goalDefeatedAt` already does permanently once the goal's been cleared, see `ConversationView`'s own `isOnBattleCooldown` check under "Losing locks that battle out for a while" below) or, once `goalDefeatedAt` is actually set, a one-off "♻️" reset button in that same slot instead - the two are mutually exclusive on the exact same condition, so clearing the goal effectively "replaces" the toggle with the reset button, matching how the cooldown itself is waived forever from that point on. "♻️" calls `gameStore.resetGoalDefeatedAt()` to null it back out directly - bypassing `recordGoalWin`'s first-write-wins guard so the whole "已通關" flow (跳過戰鬥 button, 首次通關 date, the map's rising hearts + neon "40") can be replayed from scratch without juggling a separate save-state key.

For the goal battle specifically (`isGoalEncounter`, `activeMonsterId` stays `null`), `Battle` swaps in `GOAL_NAME`/`goalSprite.png` wherever it would otherwise read `MONSTERS[activeMonsterId]`, and a win calls `gameStore.recordGoalWin()` instead of `captureMonster` (same once-the-fade-finishes timing). `recordGoalWin` (`data/goalEncounter.ts`) is first-write-wins, persisting the ISO date of the player's *first* win as `gameStore.goalDefeatedAt` - never overwritten by a later rematch. Right before calling it, that same effect also calls `flowStore.setIsFirstGoalWin(goalDefeatedAt === null)` - a snapshot of whether this particular win is the very first one, taken deliberately *before* `recordGoalWin` makes `goalDefeatedAt` non-null (the effect's own dependency array omits `goalDefeatedAt` for exactly this reason - see its `eslint-disable-next-line react-hooks/exhaustive-deps`). Once `goalDefeatedAt` is set, two things change on future goal battles: a real (non-dev) "跳過戰鬥" button joins the action bar next to 逃跑, dealing lethal damage through the same `damageWild` path as the dev shortcuts (so a skip still plays out the normal win pause/fade/finale, just without having to fight); and the action bar shows the recorded first-win date near the button row. `ConversationView` also shows its own "跳過對話" button (`.skipConversationButton`, top-right corner of the dialog box) throughout the finale conversation whenever `isFirstGoalWin` is `false` - tapping it runs the exact same `setPosition`-then-`endEncounter` logic `advance()`'s own terminal branch does (`enterHouseIfWinner`, shared by both), just without having to tap through every page first. It's gated on `isFirstGoalWin` rather than `goalDefeatedAt` specifically so the player's very first, one-time viewing of the finale always plays out in full - `goalDefeatedAt` alone can't tell "first win" apart from "any win", since it's already non-null by the time this conversation renders even on that very first win.

#### Entering a battle: the screen transition

`Game/index.tsx` no longer swaps `mode === "battle" ? <Battle/> : <map+dialog>` directly, nor crossfades that swap with a generic transition component - it hands *both* branches into `BattleTransition` as `battleContent`/`otherContent` props, and `BattleTransition` alone decides which is actually mounted at any moment (its own `displayed` state), plus renders whatever distortion overlay is currently playing on top of that. This ownership is what lets the map->battle swap happen completely hidden - there's no independent opacity crossfade left anywhere to accidentally show the old screen fading through mid-transition.

On every edge except entering battle (i.e. leaving battle back to `"conversation"`, or any other `mode` change), `displayed` just tracks `mode` directly and `phase` stays `"idle"` - an instant swap, no distortion, no fade. Only the specific `!== "battle" -> "battle"` edge triggers the full sequence: `BattleTransition` rolls one of three CSS-only distortion variants at random (`"radial"`, `"stripes"`, `"particles"` - a spinning conic-gradient wedge pattern, horizontal repeating-linear-gradient bands, and a repeating radial-gradient dot field, respectively) and steps through four timed phases, each just a `setPhase` off a `setTimeout` chain, while `displayed` stays `"other"` (the map/dialog, still fully opaque and interactive-looking) throughout the first three: `"freeze"` (`FREEZE_MS`, 300ms - the overlay is already mounted and `pointer-events: auto`, so the map/conversation screen genuinely can't be clicked again, even though nothing's visibly different yet), `"flash"` (`FLASH_MS`, 360ms - a plain white div pulsing 0→1→0 opacity), `"cover"` (`DISTORT_IN_MS`, 900ms - the rolled variant's pattern grows from a small rotated dot at screen center out to fully covering the overlay, `animation-fill-mode: both` holding that fully-covered frame once the growth finishes). (These are intentionally slowed to ~3x a natural transition's pace, at the user's own request - not a mistake to "fix" back down.) Only once `"cover"` actually finishes does `displayed` flip to `"battle"` (mounting a fresh `Battle` instance) in the exact same `setTimeout` callback that starts `"reveal"` (`DISTORT_OUT_MS`, 600ms - the same pattern keeps expanding *past* full coverage while fading out, reading as flying past the camera rather than retreating back into its own center) - the swap and the start of the reveal are the same instant, so `Battle` only ever mounts already hidden behind full opaque coverage, never mid-fade.

`Battle` never reads `BattleTransition`'s phase or even knows it exists at runtime - it just imports `DISTORT_OUT_MS` (the "reveal" stage's own duration - the only stage still running by the time `Battle` exists at all) and delays its own one-time entrance sequence (`isEntering`, true from mount for exactly `ENTRANCE_LOCK_MS`) to start counting from its own mount, offset by that one remaining duration. The enemy sprite's `.enemyEnter` (slide+scale in from the right) starts right at `DISTORT_OUT_MS` after mount - i.e. exactly when the overlay has fully cleared; the player's `.playerEnter` (mirrored, from the left) starts `ENTER_PLAYER_DELAY_MS` (360ms) later, so the enemy visibly lands first, player following a beat behind. `.enemyInfo`/`.playerInfo` (the HP blocks) each get the same `--enter-delay`/`--enter-duration` as their own sprite via `.infoEnter` - a plain opacity fade-in (not a slide, since they're pinned to a fixed corner) timed to finish exactly alongside the sprite they belong to, rather than just sitting there fully visible from the start. All of these durations/delays are passed in as inline `--enter-delay`/`--enter-duration` custom properties (`Battle/index.tsx`), not hardcoded in the CSS, so a future change to any of `timing.ts`'s constants can't quietly drift out of sync with what the keyframes actually play. Once both sprites have landed, `.introBanner` ("遭遇了 X！") fades in over `INTRO_BANNER_FADE_MS` (600ms), holds for `INTRO_BANNER_HOLD_MS` (2400ms), then fades back out over the same 600ms. `.actionBarLocked` dims the *entire* action bar (escape/skip/dev buttons, the whole attack grid) to 40% opacity and `pointer-events: none` as one unit for as long as `isEntering` is true, rather than disabling each button individually - `handleAttack` also short-circuits on `isEntering` itself as a defense-in-depth backstop, same pattern as its existing `pendingOutcome` guard, in case anything ever bypasses the disabled/pointer-events wrapper. The wild monster's own attack clock (`nextWildAttackAtRef`) is likewise initialized `ENTRANCE_LOCK_MS` later than a plain `Date.now() + WILD_ATTACK_INTERVAL_MS` would be - its first attack (and thus every attack after it, since each just adds `WILD_ATTACK_INTERVAL_MS` to the last) only starts counting down once the whole entrance sequence (and thus the action bar) has actually unlocked, rather than ticking away silently while the player still can't act back.

#### Losing locks that battle out for a while

A monster loss (not an escape, and not the goal battle - see below) sets `gameStore.battleCooldowns[monsterId]` to `Date.now() + BATTLE_LOSS_COOLDOWN_MS` (`data/monsters/battleFormulas.ts`, 5 minutes) - set in `Battle/index.tsx`'s same pending-outcome effect that calls `captureMonster` on a win, right as `concludeBattle("lose")` fires. This is a separate `Record<string, number>` from `gameStore.cooldowns` (the in-battle per-attack cooldown) - different concern, different key space (this one is never touched mid-battle, only on the way out). While that timestamp hasn't passed, walking up to that same monster and talking to it shows `buildCooldownConversation` (`data/conversations/engine.ts`) instead of the normal script - `ConversationView` picks this over `CONVERSATIONS[activeMonsterId]` whenever `battleCooldowns` says so, but only for a *fresh* encounter (`battleOutcome === null`) - the just-finished battle's own outcome recap still always shows first, cooldown or not. It's two pages, not one: the first is `openingPage`, the exact same kickoff page an unlocked encounter would open with (`CONVERSATIONS[activeMonsterId][0]`) - `ConversationView` passes it in rather than `buildCooldownConversation` inventing its own, so a locked encounter still opens with the same beat as an unlocked one. The second (terminal, `"end"`, no way to reach `enter_challenge`) is the monster's own refusal in its own voice (`speaker: "monster"`), naming how much longer is left (`formatCooldownRemaining`, "X 分鐘" - whole minutes only, rounded up). The remaining-time text is a one-off snapshot taken from `ConversationView`'s own render at the moment the player enters this page, not a live countdown - `Dialog`'s pagination effect only re-runs on `pageIndex`/`activeMonsterId`/`isGoalEncounter`/`battleOutcome` changing, so it never re-paginates just because the underlying number ticks down on a later render. One exception waives it entirely regardless of any of the above: once `gameStore.goalDefeatedAt` is set (the goal's been cleared at least once, see above), this cooldown stops applying to every monster immediately - `ConversationView`'s own check is gated on `goalDefeatedAt === null`, so a still-pending cooldown timestamp left over from before the goal was cleared just stops mattering rather than needing to be individually cleared. Before that point, a dev-only `flowStore.devCooldownLockDisabled` toggle (the map screen's "🔓" icon, see the dev-tools paragraph above) can bypass the same check for testing, without needing to actually clear the goal.

The goal battle itself is exempt from this lock entirely, on top of that - it's tough enough already without also punishing a loss with a wait. `Battle/index.tsx` never calls `setBattleCooldown` at all when `isGoalEncounter` is true (a monster loss still does), and `ConversationView`'s own `isOnBattleCooldown` check short-circuits on `!isGoalEncounter` first, so even a `battleCooldowns.goal` entry saved from before this exemption existed can't still lock it out. Losing the goal battle still shows `buildGoalLossConversation`'s own strategy tip either way (see below) - only the lockout itself is gone.

#### Goal battle: the coldnoodle self-heal

A boss mechanic exclusive to the goal battle (`isGoalEncounter`, gated in the same wild-attack `setInterval` effect that spits at the player every `WILD_ATTACK_INTERVAL_MS`): `Battle/index.tsx` keeps its own `wildSpitCountRef`, incremented on every one of those spits, and every `GOAL_SELF_HEAL_INTERVAL_SPITS`-th one (`data/monsters/battleFormulas.ts`, currently 3) queues a follow-up right after that spit lands. Rather than being thrown across the battlefield like a captured monster's icon, the coldnoodle appears beside the goal's own sprite like a side dish - `.coldNoodleSideDish` in `Battle/styles.css` shifts/fades it in over `COLD_NOODLE_ENTER_MS`, holds it wiggling in place for `COLD_NOODLE_WIGGLE_MS` (a separate `rotate`-only animation looping a fixed iteration count, kept independent from the shift/fade's own `translate`+`opacity` animation since `rotate`/`translate` are their own standalone CSS properties, not sub-parts of the single `transform` shorthand two animations would otherwise fight over), then fades back out in place over `COLD_NOODLE_EXIT_MS` without moving again. Right as the wiggle ends (`COLD_NOODLE_HEAL_DELAY_MS` after the spit lands), the same build/hold/release glow a healer's own heal uses kicks in (`useHealEffect`, a second instance for the enemy sprite - `.enemyHeal`/`enemy-heal` in `Battle/styles.css`, an exact duplicate of `.playerHeal`/`player-heal` since nothing else ever animates the enemy sprite's `filter`), and only once that glow finishes does `flowStore.healWild` actually restore `GOAL_SELF_HEAL_PERCENT` (10%) of `wildMaxHp` - a fixed amount since `wildMaxHp` never changes mid-battle. `healWild` mirrors `healProtagonist`'s own clamp (`Math.min(state.wildMaxHp, state.wildHp + amount)`). A `familyToast` banner (the same generic one the attack grid's family throws use) announces it. No other wild monster ever heals itself - this whole path is gated on `isGoalEncounter` alone.

#### Goal battle: a hint after losing

Since the goal is deliberately tougher than a regular monster (see the coldnoodle self-heal above), losing to it specifically shows one extra, terminal page the generic `buildOutcomeConversation("lose")` reaction doesn't have. `buildGoalLossConversation(GOAL_NAME)` (`data/conversations/engine.ts`) keeps the same first page every defeat gets ("唔...小風被X打倒了...") but follows it with a tip spoken by the goal itself (`speaker: "monster"`, `"end"`): chaining same-family attackers *and* healers alike, both step-scaled so a full chain deals/heals nearly 50% more than flat individual hits would (see "Family-adjacency attack bonuses" below). `ConversationView` picks this over the generic reaction only when `isGoalEncounter && battleOutcome === "lose"` - a regular monster loss, or escaping the goal battle, still gets the plain generic text.

### Family-adjacency attack bonuses

Every monster carries a battle-adjacency `attackFamily`/`attackStep`. `attackFamily` is a curated field baked into `monsters.generated.json` (6 groups: 5 themed "情人節" (Valentine) release-batch categories - `動物情人節`/`屬性情人節`/`星座情人節`/`植物情人節`/`水果情人節` - plus `小X媽` for the 2 healers, 小風媽/小馬媽) - not derived from the name by any rule in code (`data/monsters/attackFamily.ts`'s `computeAttackFamily(name)` explores a from-the-name alternative, but it's unused at runtime, only exercised by its own unit test). `attackStep` is a flat rule instead, computed in `monsters.ts`: `1` for healers, `0.1` for everyone else. The innate attack has no family (`null`) and never groups with a neighbor.

The battle attack line (`Battle/index.tsx`) starts as the innate attack first, then `gameStore.monsterOrder` (see below) turned into buttons - local (unpersisted) state from there, since the innate attack itself joins the same reorder cycle as any captured monster (see below) but its own position is never saved. `Battle/attackGroups.ts`'s `groupByAdjacentFamily` splits that line into maximal runs of consecutive same-family members - each run is one "linked" glow box (`.attackButtonLinked`, a per-button `inset` box-shadow/background driven by `hueForFamily`; every captured-monster attack also always shows a small corner `.familyDot` in this same hue regardless of cooldown, reading the option's *true* family rather than the cooldown-neutralized one grouping uses, so a player can plan adjacency ahead of time. `hueForFamily(family, allFamilies)` spaces hues evenly across `allFamilies` - a stable, deduplicated, sorted list of every `attackFamily` that actually exists across `MONSTERS`, computed once in `Battle/index.tsx` as `ALL_FAMILIES` - rather than hashing the name directly, which can't guarantee two arbitrary families won't land on similar-looking hues by chance. `.attackGroup`'s wrapper is `display: contents` so grouping never changes the row's own gap/alignment - a cooling-down neighbor is treated as family-less for this computation, since it can't actually be thrown right now and would otherwise wrongly extend the chain). Tapping any member throws the whole run together: the *k*-th member (0-indexed, left-to-right within its own run) contributes `1 + k * step` instead of a flat 1 - so a run of 3 non-healers (step 0.1) deals `1 + 1.1 + 1.2 = 3.3` total, split across however many of the run's members are healers (heal that much, scaled the same way) vs. attackers (damage the wild monster) - currently only the `小X媽` family (小風媽/小馬媽) are healers, at the higher step 1. A real (2+ member) attack throw also launches each member `GROUP_THROW_STAGGER_MS` (300ms) after the last, so the volley reads as distinguishable throws rather than one blob, and shows a callout ("X 系列攻擊，效果卓越") centered within `.battlefield` (the fight area, not the action bar/attack grid below it) for exactly as long as that whole throw takes to land. Unlike healing (below), each attacker's own damage lands the instant *its own* throw arrives (with its own hit-bump on the enemy sprite) rather than being bundled into one combined hit once the last member lands - a run of N attackers reads as N separate hits, each its own `1 + k * step` amount. Healers throw the same staggered way - each healer's own icon is thrown at the player rather than the wild monster (`getTrajectory("selfPlayer")` - the player's own center for both ends, a self-toss rather than a cross-battlefield shot; `throw-arc`'s own keyframes still apply a fixed vertical arc percentage regardless of horizontal distance, so an identical from/to still reads as a real toss-up-and-catch; `.thrownIconSelf`/`throw-arc-self` swap out just the `animation-name` to hold `scale(1)` constant throughout rather than `throw-arc`'s own 1.3 -> 0.8 -> 0.3 near-to-far shrink, since a self-toss never actually flies into the distance); only once every one of them has actually landed does the heal's own glow (`HEAL_ANIMATION_MS`, 3s - a brightness flash that builds, holds, then releases, well beyond the quick 300ms attack/hit flash `useSpriteEffect` normally uses) begin, and `healProtagonist` only actually applies once that whole glow finishes, not instantly. The callout ("X 進行治療！" solo, "小X媽 系列治療，效果卓越！" grouped) spans the throws-plus-glow combined.

`gameStore.monsterOrder` (front-to-back monster ids, captured monsters only - never the innate attack) is the single persisted source of both the duckling follower trail (`Maze/index.tsx`) and the captured-monster portion of the battle line above - a battle's own reordering carries over to the map and into the next battle, rather than resetting each time. A new capture prepends its id (`followerTrail.ts`'s `addToOrder`, called from `gameStore.captureMonster`); a save from before this field existed falls back to `orderByMostRecentlyCaptured(captured)` on hydrate. Whenever a group is thrown (innate included), each of its members - not the group as one block - alternates between the back and the front of the *whole* line, continuing the same back/front/back/... cycle across every throw in the battle (`Battle/index.tsx`'s `nextPlacement`) - this is deliberate "shuffle": a repeated group throw scatters rather than just gluing itself back together at one end every time. Only the captured-monster portion of the resulting order is what actually syncs out to `reorderMonsters` - the innate attack's own position resets to the front every fresh battle. Visually, a thrown button narrows its actual width/padding/border down to 0 (not just a cosmetic transform) before the reorder applies, then grows back the same way at its new spot - so the real flex layout reflows and every other button visibly slides to fill the gap, rather than just fading in place. Every captured-monster attack also always shows a small corner `.familyDot` (same hue as its glow) regardless of cooldown, so a player can plan adjacency even while an attack is still cooling down - it reads the option's *true* family, never the cooldown-neutralized one grouping itself uses.

### Deploy date label

`src/scale.ts` aside, the other build-time injection is `__DEPLOY_DATE__` — a webpack `DefinePlugin` value set to `DEPLOY_DATE=$(date "+%Y-%m-%d %H:%M")` by the deploy script. Shown as an "advanced info" field on the `/settings` page rather than on the map screen. Empty in production if not set; shows `1970-01-01 00:00` stub in local dev.

### Testing a store-driven component

Most tests here target pure functions extracted into their own file (`houseState.ts`, `followerTrail.ts`, `passability.ts`, ...) - the established convention, since it needs no React/DOM scaffolding at all. `Maze/index.test.tsx` is the one exception, actually rendering `Maze`/`MazeContainer` (via `react-test-renderer`) to cover behavior that lives inline in the component (the teleport-triggered effects), not in an extractable pure function. Getting a real component test running here needed two things `jest.config.js` didn't have:

- **Binary/CSS/text imports**: `Maze/index.tsx` pulls in `.png`/`.css`/`.txt` files directly, none of which plain Jest can transform. `jest/fileMock.js` (pngs -> a fixed string), `jest/cssMock.js` (css -> a `Proxy` returning each class's own source name, e.g. `styles.followerWrap` -> `"followerWrap"` - critical so tests can find elements by their real class rather than a webpack hash), and `jest/rawTransform.js` (map.txt -> its actual raw text, so a test exercises the real map layout) all wire in via `jest.config.js`'s `moduleNameMapper`/`transform`. The CSS mock needs `__esModule: true` on its export - without it, Babel's default-import interop reads a `.default` property off the mock itself, and since that's a property access too, the `Proxy`'s own `get` trap intercepts it and returns the string `"default"` instead of the `Proxy`, collapsing the whole mock to one wrong value.
- **jsdom, per-file**: zustand's React binding (`use-sync-external-store`'s shim) detects `typeof window === "undefined"` and silently falls back to an SSR-style implementation that calls `getSnapshot()` once and never subscribes at all - under this project's usual `testEnvironment: "node"`, a store-driven component renders once and then never reacts to a later store update, no matter how correct the component's own logic is. A `/** @jest-environment jsdom */` docblock at the top of `Maze/index.test.tsx` opts just that file into a real `window`, without changing the global config other tests rely on.

## Deployment (Google Apps Script)

Deployed via clasp as a standalone web app (not gh-pages). React source lives in `src/`; production build outputs inlined `gas/index.html` alongside `gas/Code.js` (doGet handler).

```sh
npm install
npm run login          # authenticate with Google
npm run setup          # first time only — creates GAS project, writes scriptId to .clasp.json
npm run deploy         # build, push, and activate the live deployment
```

| Command | Description |
|---|---|
| `npm run pull` | Pull latest scripts from GAS editor |
| `npm run push` | Build, push to GAS (also pushes git) |
| `npm run open` | Open GAS editor in browser |

## Google Sheet integration

The GAS project is container-bound to a Google Sheet, but every save lives specifically on the sheet named `"2026"` within it (`Code.js`'s `STATE_SHEET_NAME`/`getStateSheet()`) - not just whichever sheet happened to be active, since the container-bound spreadsheet holds other sheets too. Storage is one row per save-state key: column A = key, column B = the full state JSON blob (position, captured monsters, monsterOrder, cooldowns, goalDefeatedAt, timestamp — see `store/types.ts`). `google.script.run.saveState(key, json)` finds the key's row and overwrites column B (appending a new row if the key is new); `loadState(key)` reads it back; `listStateKeys()` returns every non-empty key across the whole sheet (backs Settings' dev-only key-picker, see below). There's no more per-field (x/y) column layout — the whole persisted slice travels as one JSON string.
